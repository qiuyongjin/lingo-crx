# History Panel: Side Panel Migration

**Date:** 2026-06-12
**Status:** Approved

## Overview

Migrate the history panel from a fixed-position drawer inside the content script Shadow DOM to the Chrome Side Panel API (`chrome.sidePanel`). This eliminates page content occlusion — the browser natively shrinks the viewport so page content reflows correctly, including `100vw` elements and `position: fixed` elements.

## Motivation

The current fixed-position drawer (320px wide, full viewport height) overlays page content on the right side of the viewport. Any page content in that zone is completely hidden. While glassmorphism or resizing could mitigate the problem, only the Side Panel API achieves zero occlusion — the browser itself allocates space for the panel and shrinks the page viewport accordingly.

## Architecture

```
                          Before                                        After

content script Shadow DOM                     content script Shadow DOM          side panel (independent context)
├── FloatingButton ←→ HistoryApp              └── FloatingButton                  ├── sidepanel/App.tsx
└── HistoryPanel (fixed drawer + overlay)           │                             │   ├── useHistory() hook
        ├── useHistory() hook                       │ sendMessage                 │   │   └── onChanged listener
        └── chrome.storage.local                    ▼                             │   └── HistoryPanel.tsx
                                              background.ts                       └── chrome.storage.local (read + listen)
                                              └── chrome.sidePanel.open({tabId})       ▲
                                                                                       │
                                                                               content.tsx (on word lookup)
                                                                               └── historyStorage.add() (unchanged)
```

**Key separation:** The FloatingButton lives in the content script (page context). The HistoryPanel lives in the side panel (browser context). They communicate via `chrome.runtime.sendMessage` and share state through `chrome.storage.local`.

## Data Flow

1. **Word lookup** → content script writes to `chrome.storage.local` (unchanged from current)
2. **Side panel mounts** → reads full history from `chrome.storage.local`, then listens to `chrome.storage.onChanged` for incremental updates
3. **FAB click** → `chrome.runtime.sendMessage({ type: "openSidePanel" })` → background calls `chrome.sidePanel.open({ tabId })`
4. **Panel close** → browser-native close button; no custom logic needed

## Side Panel Entrypoint

### Route

The side panel is **not tab-specific** — path is always `sidepanel.html`. All state comes from storage. When a tab doesn't support side panels (e.g., `chrome://` pages), the browser grays out the toolbar button; the FAB on those pages simply does nothing (no-op on `sendMessage` error).

### `entrypoints/sidepanel/`

```
entrypoints/sidepanel/
├── index.html           — WXT HTML entry, mounts React root
├── App.tsx              — root component: useHistory + render HistoryPanel
└── (tsconfig, etc. — auto-generated by WXT)
```

`App.tsx`:
```tsx
function App() {
  const { items, loading } = useHistory(); // now uses onChanged

  return (
    <div className="lingo-sidepanel">
      <HistoryPanel
        items={items}
        loading={loading}
        onClose={null} // no close button in side panel — browser handles it
      />
    </div>
  );
}
```

**No close button, no overlay, no backdrop.** The browser provides the close mechanism (X button on the side panel header).

## FloatingButton Behavior

- **Always visible** — no longer hidden when panel is open (no occlusion, no reason to hide)
- **Click** → `chrome.runtime.sendMessage({ type: "openSidePanel" })` — opens or focuses the side panel
- **Drag-to-reposition** unchanged, persisted to `chrome.storage.local`
- **On pages where side panel is unavailable** (e.g., `chrome://extensions`), the `sendMessage` call fails silently; FAB is harmless

## useHistory Hook Update

```ts
// Before (content script context):
useEffect(() => {
  if (isOpen) reload();
}, [isOpen]);

// After (side panel context):
useEffect(() => {
  const listener = (changes: Record<string, chrome.storage.StorageChange>) => {
    if (changes["lingo-history"]) reload();
  };
  chrome.storage.onChanged.addListener(listener);
  return () => chrome.storage.onChanged.removeListener(listener);
}, []);
```

This ensures new lookups appear instantly in the side panel without requiring close/reopen.

## Edge Cases

- **Panel already open**: `sidePanel.open()` focuses the existing panel rather than opening a duplicate (browser-native behavior)
- **Tab doesn't support side panel**: `chrome.sidePanel.open()` may reject; catch silently. FAB does nothing.
- **Storage cleared externally**: `onChanged` fires → panel re-renders with empty state
- **Multiple tabs**: each tab's FAB opens the same side panel for its own tab. Side panel content is tab-independent — it always shows the shared history from storage.
- **History overflow**: 500-item cap and LRU eviction unchanged from current implementation
- **Side panel width**: browser default is ~320px; user can drag the separator to resize (built-in behavior, no code needed)

## Permissions

Add `"sidePanel"` to `wxt.config.ts` manifest permissions. This is the only configuration change needed — WXT auto-generates the side panel entrypoint.
